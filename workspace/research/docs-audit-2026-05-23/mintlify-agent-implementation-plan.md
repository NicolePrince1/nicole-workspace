# Mintlify agent implementation plan — Oviond docs

This extends the launch QA with Mintlify-specific operating rules for making `docs.oviond.com` excellent for humans and AI agents.

## Updated verdict

The docs cleanup should happen as a **Mintlify docs-as-code workflow**, not as ad hoc page rewrites.

The right system is:

1. Install Mintlify-native agent context in the docs repo.
2. Add Oviond-specific writing and terminology rules.
3. Fix the P0/P1 launch blockers as PR-ready changes.
4. Validate with Mintlify CLI and a separate reviewer pass.
5. Keep the maintenance loop tied to product/API changes.

## Agent workflow

Use separate sessions/agents instead of one agent doing everything.

### Research agent

Reads:

- existing docs pages
- `docs.json`
- OpenAPI spec
- `llms.txt`
- product/API source truth
- support tickets/Gleap patterns where relevant

Output:

- page-level facts
- duplicate/gap list
- source-of-truth references
- risk notes

### Writer agent

Rules:

- read 2–3 similar pages before editing
- update existing pages before creating new pages
- use Mintlify components, not raw HTML
- keep pages focused and short enough for AI agents
- produce PR-ready diffs

### Reviewer agent

Checks:

- terminology consistency
- `CLAUDE.md` compliance
- passive voice / run-ons / AI slop
- duplicate pages or sections
- incorrect component usage
- internal/vendor leaks
- mismatched API references

### Validator agent

Runs:

```bash
# exact command depends on the installed Mintlify CLI; verify with --help first
mintlify dev
mintlify validate
```

If the project uses the newer `mint` binary, use:

```bash
mint dev
mint validate
```

Validation gate:

- Mintlify local preview runs
- no MDX/component errors
- no broken navigation references
- OpenAPI spec validates
- no broken internal links
- no public sample content
- generated `llms.txt` remains clean

## Iterative workflow

Use this loop for every page or section:

1. **Generate** — draft or update with source references.
2. **Check** — separate reviewer checks style, facts, and consistency.
3. **Revise** — writer applies review fixes.
4. **Validate** — run Mintlify CLI and link/spec checks.
5. **PR** — final output is a diff with rationale and QA notes.

Do not merge docs generated by a single pass without a reviewer pass.

## Mintlify component rules

Add this decision table to `CLAUDE.md`.

| Need | Use | Avoid |
| --- | --- | --- |
| Sequential instructions | `<Steps>` and `<Step>` | raw numbered paragraphs for long flows |
| Warning about destructive/risky action | `<Warning>` | bold-only warnings |
| Helpful caveat | `<Note>` | parenthetical caveats buried in body copy |
| Best-practice tip | `<Tip>` | generic “Pro tip” text |
| Page options/navigation | `<CardGroup>` and `<Card>` | long link lists |
| API params | `ParamField` / OpenAPI-generated params | markdown tables when Mintlify supports param components |
| Response fields | `ResponseField` / OpenAPI schema | prose-only schema descriptions |
| Code examples | fenced code blocks with language | unfenced inline pseudo-code |
| Tabs/language variants | `Tabs` | repeated sections with duplicate headings |
| Reusable product definitions | snippets/includes | copy-paste drift |

## Severity model for PRs

### Critical

Blocks launch or damages trust immediately.

Examples:

- sample Plant Store spec live
- invalid OpenAPI path templates
- docs build fails
- broken primary nav
- auth examples contradict actual API
- destructive action docs are misleading

### High

Should fix before launch.

Examples:

- internal implementation leaks
- thin core onboarding/API pages
- inconsistent product terminology
- billing/pricing uncertainty
- MCP support overclaims
- missing API request/response/error examples

### Medium

Can ship shortly after launch but should not become permanent debt.

Examples:

- duplicate generic H1s
- minor grammar/readability issues
- weak cross-links
- pages without screenshots/GIFs
- boilerplate visible only in markdown output

## Writing/readability enforcement

Add this to `CLAUDE.md` and reviewer prompts:

- Use second person: “you”.
- Use active voice.
- Keep paragraphs to **two sentences max**.
- Prefer short pages over long catch-all pages.
- One concept per page.
- Lead with what/why before how.
- Put prerequisites before steps.
- Sentence case for headings.
- Do not use: “simply”, “just”, “easily”, “obviously”, “leverage”, “robust”, “seamless”, “delve”, “game-changing”.
- Do not expose internal stack/vendor names unless the user must configure that vendor directly.
- Use exact product terms from the glossary.

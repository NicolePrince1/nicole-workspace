## Stripe skill architecture diagnosis + fix

- Chris asked for the Stripe skill to be fixed properly and said current real MRR is `$28,712`.
- Research conclusion:
  - Stripe MCP is not the main fix for this problem; it can help with interactive access, but it does not solve MRR accuracy by itself.
  - the better pattern is: fast operational API reads + a canonical invoice-backed MRR path + local caching.
- Root cause found in the old Stripe skill:
  - the old MRR path relied too heavily on subscription price math
  - archived/tiered prices were being undercounted or modeled badly
  - that was why the old skill could show nonsense like roughly `$17k` MRR
- Additional diagnostic finding:
  - simply fixing tier expansion on raw price math swung too far the other way (around `$38k`), proving pure price math was still the wrong architecture for Oviond
- Implemented fix in `skills/stripe-reader/scripts/stripe_analytics.py`:
  - canonical MRR now uses latest-invoice recurring subscription lines normalized to monthly
  - line-level discounts are accounted for where present
  - revenue statuses included by default: `active`, `past_due`, `unpaid`
  - fallback order is now:
    1. latest invoice recurring lines
    2. current subscription-item pricing with archived-price + tier expansion support
    3. latest paid recurring invoice
  - added local cached snapshots under `/data/.openclaw/db/stripe-reader/`
  - added `--live` and `--cache-ttl` support so fast operational checks are instant unless forced live
  - optimized revenue mode to only read the needed invoice date windows
- Validation result after the rewrite:
  - new live MRR output lands in the expected band near the dashboard truth (`$28,712`), rather than the broken old `$17k` estimate
  - cached reads are now effectively instant for repeat checks
- Documentation updated:
  - `skills/stripe-reader/SKILL.md`
  - `skills/stripe-reader/references/metrics.md`

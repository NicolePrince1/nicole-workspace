## Telegram voice-note regression root cause was narrowed further than the earlier normalizer hypothesis

- Later live forensics on `/data/.openclaw/agents/main/sessions/5d1c451e-ffb7-4abe-8ae4-abc7b4cc161a.jsonl` showed the failing turn sequence was: `message` tool sends the text reply to the same Telegram chat, `tts` succeeds and generates `/tmp/openclaw/tts-Dtaolg/voice-1776165649057.opus`, the TTS tool result includes `audioAsVoice: true`, and the assistant then ends with `NO_REPLY`.
- That evidence means TTS generation and voice-note compatibility were both still working in the failing session; the loss happens after TTS, not before it.
- The stronger root-cause candidate is the reply-suppression path in `/app/node_modules/openclaw/dist/reply-payloads-dedupe-OtM2z7YX.js` and `/app/node_modules/openclaw/dist/agent-runner.runtime-CUJd1OG7.js`: `shouldSuppressMessagingToolReplies(...)` plus `buildReplyPayloads(...)` can zero out the entire final reply payload set when a `message` tool send already targeted the same chat.
- Practical implication: when the model follows the pattern `message` -> `tts` -> `NO_REPLY`, OpenClaw may suppress the queued TTS media payload along with duplicate text, which cleanly explains why Chris still receives text but no voice note.
- Treat this suppression logic as the best current root cause for the Telegram voice-note regression. The earlier `normalizeOutboundReplyPayload()` hypothesis should be considered superseded unless a patch/retest disproves the suppression finding.

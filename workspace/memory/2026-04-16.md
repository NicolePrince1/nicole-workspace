## Supabase-first tracking architecture was chosen as the clean rebuild path

- Chris asked whether Supabase Cloud already had an easier built-in way to handle Oviond’s tracking and lifecycle architecture.
- Nicole checked the real Supabase feature/docs surface and concluded there is no single magic plugin, but Supabase Cloud already provides most of the primitives needed: Postgres extensions, Database Webhooks built on `pg_net`, Auth Hooks, Edge Functions, and Queues.
- The durable architectural call was: Supabase/backend should own truth, Sequenzy should be the lifecycle brain rather than the truth engine, and ad platforms should be mirrors/consumers of canonical backend events.
- Nicole produced and shared a full Google Sheet architecture spec at `https://docs.google.com/spreadsheets/d/1xjO-OajleRRO22fnJsqP2VummoV6Zgacu4A1QkYiDWk/edit?ouid=107370502050260479878`.
- The clean minimal build was defined as canonical Supabase tables for users, workspaces, subscriptions, event log, audience memberships, and attribution sessions; a small set of important state-change triggers/webhooks; one central Edge Function event router; and retry handling via Supabase Queues.
- Canonical server-side events for the rebuild were defined as: `user_created`, `email_verified`, `workspace_created`, `trial_started`, `activation_completed`, `pql_reached`, `invoice_paid_first`, `subscription_past_due`, `subscription_canceled`, `trial_expired`, and `won_back`.
- Audience logic was defined centrally in Supabase-derived lifecycle/audience tables, then mirrored outward to Sequenzy plus ad platforms.
- The exact Sequenzy audience set proposed for the architecture was:
  - `lead_prospect`
  - `signup_started_no_user`
  - `trial_started_all`
  - `trial_unverified`
  - `trial_verified_not_activated`
  - `trial_activated_not_paid`
  - `trial_pql`
  - `trial_expiring_3d_unactivated`
  - `trial_expiring_3d_activated`
  - `trial_expired_0_14d`
  - `trial_expired_15_60d`
  - `paid_new_30d`
  - `paid_active_monthly`
  - `paid_active_annual`
  - `past_due`
  - `churned_0_30d`
  - `churned_31_180d`
- Durable ad-platform rule: paid users must always be excluded from acquisition, current trials excluded from top-of-funnel prospecting, and only sufficiently large CRM audiences mirrored for retargeting/winback.
- Durable attribution rule: persist `anonymous_id`, UTMs, `gclid/gbraid/wbraid`, `fbclid/fbc/fbp`, `li_fat_id`, and `twclid` before signup, then bind them to the Supabase user at signup or trial start, because missing that bind permanently wrecks attribution quality.
- Durable implementation recommendation: treat the sheet as the implementation spec, build the canonical event log plus lifecycle/audience tables first, wire Sequenzy and platform sync off that truth layer, and do not rely blindly on Sequenzy’s native Supabase integration unless it proves it can mirror Oviond’s exact event model and audience flags.

- Deep doc read conclusion on 2026-04-16: Nicole's current recommendation is to test `Placid` first, not Bannerbear. Main reasons: Placid exposes the cleanest end-to-end hosted-agent path for this runtime, including project-scoped bearer auth, explicit template CRUD, collections, direct media upload, deterministic image/PDF/video generation, storage transfer options, a simple signed URL API, an embeddable Editor SDK, and an MCP explicitly designed for AI workflows. Bannerbear is strong and more feature-rich in some areas, but its model feels slightly more indirect for Nicole's likely first implementation because it leans more on external media URLs, editor sessions, signed-base mechanics, and a broader platform surface that is great but not as straightforward for the exact Oviond automation path.
